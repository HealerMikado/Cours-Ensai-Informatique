Le fonctionnement de git

L'arhicture générale

  • Un dépôt central (remote)
  • Plein de dépôt locaux (clones)
  • Des zones de transit
  • Des "workspaces"

Dessin

Fonctionnement

  • Clone du dépôt central (+historique)
  • Modification du code de la workspace
  • Envoie du code dans le dépôt local (via zone de transite)
  • Envoi du différentiel du dépôt local dans le dépôt distant
  • Seulement si on est à jour sur la remote

Les choses à retenir

  • Il faut être à jour avant de pousser une version
  • Plusieurs versions et historiques à un moment T
  • Remote fait foi
  • Possibilité de conflit de versions
  • Mais seulement en local

Un test donne-t-il une information 100% fiable ?

Définition

“En informatique, un test désigne une procédure de vérification partielle d'un système. Son objectif principal est d'identifier un nombre maximum de comportements problématiques du logiciel. Il permet ainsi, dès lors que les problèmes identifiés seront corrigés, d'en augmenter la qualité.

D'une manière plus générale, le test désigne toutes les activités qui consistent à rechercher des informations quant à la qualité du système afin de permettre la prise de décisions.

Un test ressemble à une expérience scientifique. Il examine une hypothèse exprimée en fonction de trois éléments : les données en entrée, l'objet à tester et les observations attendues. Cet examen est effectué sous conditions contrôlées pour pouvoir tirer des conclusions et, dans l'idéal, être reproduit ”

En résumé

  • Aucun test n'est parfait !
  • Donne seulement une information partielle de la qualité de votre application
  • Permet de détecter des "erreurs"
  • Et de les corriger si vous êtes conciencieux
  • Il y en a plusieurs types
  • Et ils doivent être reproductibles !

Pourquoi tester ? (à votre niveau)

  • Eviter les erreurs quand vous codez

Le Test Driven Developpement (TDD)

Pratique qui consiste à écrire les tests AVANT de coder

  • On commence par écrire un test (inputs, objet à tester,outputs)
  • Puis on définit les fonctions nécessaires pour lancer le test
  • On lance le test -> echec
  • On code la fonction pour une réussite minimal -> réussite
  • On met à jour la fonctions -> refactor

Les tests unitaires

Les tests unitaires

  • Vérifie qu'une méthode fait bien ce qu'elle doit faire
  • Cas nominaux (doivent fonctionner)
  • Cas d'erreur (doivent lever une erreur)
  • Cas à la marge (ça dépend)
  • Tests à la charge des développeurs

Architecture d'un test unitaire


                            def test_ma_fonction():
                                # GIVEN
                                Inputs

                                # WHEN
                                Le traitement à tester

                                # THEN
                                Des petits traitements supplémentaire comme nettoyer la base

                                assert  actual_result == expected_result

Architecture d'un test unitaire


                        def test_creer_compte():
                            # GIVEN
                            pseudo = "Iron Man"
                            mdp = "Le plus intelligent des Avengers"
                            # WHEN
                            compte = compte_service.creer_compte(pseudo, mdp)
                            # THEN
                            if compte.id: 
                                compte_dao.delete(compte) # On supprime le compte
                            assert compte.id > 0

C'est quoi un bon test unitaire ?

  • Teste qu'UNE chose
  • Le plus isolé possible
  • Un assert "simple"
  • Commenté
  • Maintenable

Programmation orientée objet

Les grands principes de la POO

Les grands principes de la POO

  • Encapsulation : un objet a des attributs, méthodes
  • Héritage : un objet peut être un cas particulier d'un autre
  • Polymorphisme : des méthodes peuvent avoir le même nom et agir différement

Classe vs instance

  • Une classe est un plan
  • Un instance est un objet construit en suivant le plan qu'offre la classe
  • Une classe est une coquille vide
  • Une instance est la matérialisation de la classe

Classe vs instance

  • Une classe peut disposer d'attributs
  • Ces attributs seront communs à toutes les instances
  • Permet de stocker certaines infos communes facilement

Exemple attribut classe


                            class Voiture:

                                vitesse_max = 500

                                def __init__(self, marque):
                                    self.marque = marque

                                def deplacement(self, vitesse):
                                    if vitesse > Voiture.vitesse_max :
                                        raise Error

Rappels sur l'héritage

Rappels sur l'héritage

  • La classe fille est un cas particulier de la classe mère
  • Possède toutes ses méthodes, attributs
  • Peut récrire les méthodes (surcharge)
  • Peut utiliser le constructeur de la classe mère via la fonction super()

Classes abstraites

Classe abstraite : définition

Une classe abstraite est une classe non intanciable directement. Offre un canvenas pour les classes qui vont en hériter. Ces classes auront directement accès aux méthodes de la classe abstraite, mais devront les redéfinir pour fonctionner.

Exemple de classe abstraite


                        class AbstractVoiture:

                            def accelerer(self):
                                raise NotImplementedError

                            def freiner(self):
                                raise NotImplementedError

                            def choc(self):
                                raise NotImplementedError

Une idée de pourquoi je m'impose ça ?

Utilité des classes abstraites pour l'extérieur

  • Interfaces entre le programme et l'extérieur
  • Noms des méthodes
  • Mais pas à ce qu'elles font
  • Car les gens n'ont pas à savoir
  • ~ endpoints d'un webservice

Utilité des classes abstraites, en interne

  • Contrat à respecter
  • Méthodes minimales à posséder
  • Lisibilité du code
  • Pattern strategy
Structure le code, le rend plus maintenable et évolutif. Mais demande plus de rigueur

Conclusion

  • Encapsulation, héritage, polymorphisme
  • Les classes abstraites permettent de structurer votre code
  • Interfaces avec l'extérieur
  • Notion fine de génie logiciel
